## v2.0.9 (2026-03-17)

### 本版本目标
- 将批量分析从串行改为可控并行，提升多文件处理吞吐。
- 批量并行参数仅由管理员通过 `.env` 配置，避免业务请求临时改写系统并发策略。
- 在 4 核 CPU / 8G 内存场景下，提供默认保守并发值，避免 CPU 长时间满载。

### 已修改功能
1. 批量分析链路改为并行执行
- `analyze_batch` 从“逐文件串行调用 `process_video`”改为基于 `ThreadPoolExecutor` 的并行执行。
- 每个文件仍独立执行完整风控与分析流程，互不影响；单文件失败不会中断其他文件任务。
- 返回结果仍保持按输入序号排序，确保前端展示顺序稳定。

2. 新增批量并行参数（支持 `.env`）
- 新增环境变量 `BATCH_ANALYZE_MAX_WORKERS`，用于控制批量分析并发度。
- 默认值设为 `2`，并带边界约束：最小 `1`，最大 `16`。
- 运行时会自动再受 `CPU 核心数` 与 `本次文件数量` 双重上限限制，防止误配导致过载。

3. 并行参数仅支持管理员配置
- `analyze_batch` 不再读取请求体中的并行度参数。
- 并发值仅来自 `.env` 的 `BATCH_ANALYZE_MAX_WORKERS`。
- 最终值统一走安全边界裁剪（文件数/CPU 核数/范围限制）。

4. 批量进度与响应补充
- 批量任务启动提示中加入“并行度”显示，便于前端/运维观察实际并发配置。
- 接口响应新增 `batch_parallel_workers` 字段，明确返回本次任务实际生效并发值。

### `.env` 当前项目值（本次更新后）
- `BATCH_ANALYZE_MAX_WORKERS=2`

### 验收结果
- Python 语法校验通过：`python -m py_compile app.py`。
- 代码检索确认关键点已生效：
  - `analyze_batch` 并行执行
  - 请求体覆盖并发参数
  - `.env` 并发参数读取
  - 响应返回实际并行度字段

## v2.0.8 (2026-03-17)

### 本版本目标
- 降低 4 核 CPU / 8G 内存场景下的风控链路 CPU 峰值。
- 将关键处理参数统一外置到 `.env`，并明确“请求体优先、缺省回退环境变量”策略。
- 保持主分析能力不降级，同时优化风控与截图阶段的资源占用。

### 已修改功能
1. 风控链路取消“默认并行预跑字幕兜底”
- `_run_video_risk_gate(...)` 移除默认 `ThreadPoolExecutor` 预跑文本兜底逻辑。
- 当前策略：主视觉风控成功则直接返回；仅当视觉风控失败/不可用且系统视觉兜底也不可用时，才触发文本兜底。
- 结果：减少一次高概率的额外 Whisper 调用，降低风控阶段 CPU 峰值。

2. 截图并发默认降至 2，并支持 `.env` 控制
- `generate_screenshots_from_steps(...)` 新增 `SCREENSHOT_MAX_WORKERS` 读取逻辑。
- 默认并发由原先固定上限策略调整为默认 `2`，非法值自动回退 `2`。
- 仍保留安全边界：并发值会被限制在 `[1, CPU 核心数]` 且不超过任务数。

3. 风控抽帧数默认调整为 4/8，并支持 `.env` 控制
- `RISK_MAX_FRAMES` 改为环境变量读取，默认值 `4`。
- `RISK_DYNAMIC_MAX_FRAMES` 改为环境变量读取，默认值 `8`，且不低于基础抽帧数。
- 保留 `RISK_MIN_FRAMES` 约束，保证风控抽帧下限与稳定性。

4. 风控链路 Whisper 模型改为 `tiny`，主分析保留 `base`
- 上传风控 agent 与系统级视觉兜底风控 agent 的 `whisper_model` 改为 `tiny`。
- 主分析流程仍按分析参数 `whisper_model` 运行（当前默认 `base`），不影响主结果质量预期。

5. 参数外置 `.env` 并统一优先级规则
- 新增/统一维护以下环境变量：
  - `WHISPER_THREADS`
  - `LONG_VIDEO_PREPROCESS_ENABLED`
  - `WHISPER_MODE`
  - `MAX_VISION`
  - `WEB_SEARCH`
- 解析策略更新为：请求体优先；请求体未传时回退到 `.env`。
- `regenerate` 接口的 `web_search` 同步采用同一优先级规则。

- `LONG_VIDEO_PREPROCESS_ENABLED=1`
- `WHISPER_MODE=base`
- `MAX_VISION=10`
- `WEB_SEARCH=0`
- `SCREENSHOT_MAX_WORKERS=2`
- `RISK_MAX_FRAMES=4`
- `RISK_DYNAMIC_MAX_FRAMES=8`

### 验收结果
- `.env` 必需参数存在性检查通过。
- 参数优先级验证通过：请求体传参时覆盖环境变量；未传时按 `.env` 回退。

## v2.0.7 (2026-03-16)

### 本版本目标
- 提升“未识别到标准步骤”场景下的结果稳定性，确保用户上传后尽量有可读产物返回。
- 统一降级与拦截结果语义，提升前后端可解释性与展示一致性。
- 优化质量分计算准确性，避免固定分值导致的评估偏差。
- 增加上传与历史数据自动保留期清理，降低磁盘长期堆积风险。
- 加固自动删除安全边界，避免路径越界导致的误删风险。

### 已修改功能
1. 新增三层保底产物机制（步骤/候选步骤/时间线摘要）
- 当标准步骤可提炼时，返回 `steps` 模式。
- 当标准步骤为空时，优先从字幕抽取“动作+对象+时间”生成 `candidate_steps`。
- 当字幕信号不足时，自动降级为 `timeline_summary`，至少输出标题、要点、时间点、置信说明与原因说明。
- 新增紧急摘要保底路径：极端失败场景下仍强制输出摘要内容，降低空结果概率。

2. 新增风控拦截说明卡语义
- 风控拦截时统一返回 `result_mode=blocked_notice`，不再输出步骤细节。
- 返回结构化说明：`risk_level / reason_code / reason / suggestions / retry_guidance`，便于前端稳定渲染。

3. 前端降级体验与文案优化
- 新增“仅生成摘要版”按钮，支持用户主动触发摘要模式。
- “未识别到步骤”不再等同失败，降级结果显示为“已完成（降级）”。
- 降级提示改为黄色提示卡，显示质量分、中文化降级原因、标题、要点、时间点与置信说明。
- 历史侧栏新增持续提醒：历史记录与总结文件仅保留 72 小时。

4. 质量分算法重构（准确性优化）
- 将质量分从“固定基线+均值”升级为多维动态评分。
- 评分维度：结构完整度、时间序列质量、置信度稳定性、来源可信度、步骤数量合理性、降级原因惩罚。
- 新增模式上限与先验权重，避免不同模式分值区间失真。
- 移除候选/摘要链路中的固定 `quality_score` 覆盖，统一走动态算法。

5. 新增 `uploads` 视频 24h 自动清理
- 新增后台任务，仅扫描并删除 `uploads` 下超过 24h 的视频文件（按文件加载时间起算）。
- 上传成功后刷新视频文件 `mtime`，确保计时从“进入系统”时刻开始。
- 非视频文件（如 `.json/.txt`）不会被删除。

6. 新增历史记录与输出目录 72h 自动清理
- 新增后台任务清理 `history.json` 中超过 72h 的记录。
- 同步清理 `outputs` 下超过 72h 且满足规则的总结目录（按 `operation_guide.md` 生成时间计算）。
- 清理任务静默运行，不弹前台打扰提示。

7. 自动删除链路新增二次边界校验（防误删）
- 上传视频自动清理在删除前新增“二次边界校验”。
- 先 `resolve(strict=False)` 再 `_assert_within(upload_root)` 校验路径归属。
- 再次确认目标是非软链接、真实文件、且扩展名在视频白名单中。
- 仅全部校验通过才执行 `unlink()`。

### 修改后的流程细节

#### A. 分析结果降级与拦截流程（更新）
1. 优先尝试标准步骤提炼，成功返回 `result_mode=steps`。
2. 若步骤为空，转为字幕候选步骤提炼，返回 `result_mode=candidate_steps`。
3. 若字幕不足，转为时间线摘要，返回 `result_mode=timeline_summary`。
4. 若命中风控拦截，返回 `result_mode=blocked_notice` 与检测说明卡。
5. 所有模式统一返回 `quality_score` 与 `degrade_reason` 供前端渲染。

#### B. 质量分计算流程（更新）
1. 按结果模式确定先验分与模式上限。
2. 计算结构、时间、置信、来源、数量五类子分。
3. 根据 `fallback_used` 与 `degrade_reason` 执行惩罚校正。
4. 归一化后输出最终 `quality_score`（0~1）。

#### C. 自动清理流程（新增）
1. 上传清理任务：周期扫描 `uploads`，仅删除超过 24h 的视频文件。
2. 历史清理任务：周期清理超过 72h 的历史记录。
3. 输出清理任务：删除不再保留且文档超过 72h 的总结目录。
4. 删除前执行路径边界校验，确保仅在允许目录内操作。

### 本次改动为达成的核心目标
- 目标 1：保证“有上传尽量有内容”，显著降低空结果与失败感知。
- 目标 2：统一后端语义与前端展示，提升结果可解释性与可审计性。
- 目标 3：提升质量分可信度，使评分更贴近真实结果质量。
- 目标 4：通过自动保留期治理控制存储增长，减少长期运维压力。
- 目标 5：通过二次边界校验强化删除安全性，降低误删风险。

### 兼容性与注意事项
- 上传24h清理仅作用于 `uploads` 下视频扩展名文件，不删除其他类型文件。
- 历史72h清理与输出目录清理以“文档时间优先、记录时间兜底”进行判定。
- `outputs` 下若存在非业务目录且含旧 `operation_guide.md`，也可能进入清理范围，建议独立隔离目录。
- 删除链路已增加二次边界校验，但仍建议保持目录权限最小化与定期备份策略。

## v2.0.6 (2026-03-16)

### 本版本目标
- 优化“安全检测阶段字幕生成”与“正式分析阶段字幕生成”的复用能力。
- 降低同一视频在上传风控与正式分析之间的重复 Whisper 转写开销。
- 在不改变原有风控策略的前提下，缩短分析前后整体处理时延。

### 已修改功能
1. 字幕缓存新增跨阶段身份标识
- `VideoAnalyzerAgent.generate_subtitles(...)` 新增 `cache_identity` 参数（可选）。
- 字幕缓存键支持使用 `cache_identity + whisper_model` 生成，避免仅依赖文件路径。
- 兼容旧逻辑：未传 `cache_identity` 时仍按原路径/大小/mtime 规则缓存。

2. 风控文本链路接入统一字幕缓存身份
- `_run_text_fallback_risk_gate(...)` 新增 `subtitle_cache_identity` 参数。
- 主风控、视觉失败兜底、视觉可用二级文本兜底均透传该身份标识。
- 上传前风控阶段使用视频 `SHA-256` 作为字幕缓存身份，便于后续复用。

3. 正式分析阶段接入同一缓存身份
- `process_video(...)` 在生成正式字幕时传入同一视频指纹作为 `cache_identity`。
- 当上传前风控已生成过字幕且模型一致时，正式分析可直接命中缓存，跳过重复转写。

4. 上传前风控链路补充传参
- `_moderate_staged_upload(...)` 与 `UploadRiskService.moderate_staged_upload(...)` 支持接收文件指纹。
- 上传预检链路在调用风控时透传指纹，确保上传阶段字幕缓存可被分析阶段复用。

### 修改后的流程细节

#### A. 上传风控字幕流程（更新）
1. 上传预检计算视频 `SHA-256`。
2. 若风控触发文本链路，会以该 `SHA-256` 作为字幕缓存身份执行转写。
3. 转写结果写入共享字幕缓存目录，供后续分析阶段直接复用。

#### B. 正式分析字幕流程（更新）
1. 分析前计算同一视频 `SHA-256`。
2. 正式字幕生成时使用相同 `cache_identity` 查询字幕缓存。
3. 命中则直接复用字幕文件，未命中才触发 Whisper 转写。

### 本次改动为达成的核心目标
- 目标 1：实现“上传前安全检测 -> 正式分析”字幕复用闭环。
- 目标 2：减少重复 Whisper 调用，提升链路吞吐与响应速度。
- 目标 3：保持模型隔离（`whisper_model` 仍参与缓存键），避免跨模型误复用。

### 兼容性与注意事项
- 仅在 `cache_identity` 一致且 `whisper_model` 一致时可命中复用。
- 若分析阶段切换到不同 Whisper 模型，系统会自动重新转写（安全且可预期）。
- 该优化不改变黑名单、风控拦截和错误码语义，仅优化字幕生成开销。

## v2.0.5 (2026-03-16)

### 本版本目标
- 优化“分析前功能检测”，重点提升上传前风控的执行效率。
- 在现有黑名单拦截基础上，新增“同文件 + 同模型配置”风控结果缓存快路径。
- 减少重复模型调用与重复哈希计算，加快上传到分析的整体处理链路。

### 已修改功能
1. 新增上传前风控结果缓存
- 新增缓存文件：`uploads/.risk_result_cache.json`。
- 新增缓存服务：`RiskResultCacheService`（含过期与容量控制）。
- 缓存键由“文件 SHA-256 + 风控模型配置签名 + 风控策略签名”组成，避免误复用。
- 仅缓存“非阻断、非基础设施异常”的风控结果，保证安全边界不被放宽。
- 默认缓存参数：`TTL=24h`、`MAX_ENTRIES=500`。

2. 黑名单服务新增“按哈希直查/直写”能力
- 新增 `match_fingerprint(...)` 与 `register_blocked_fingerprint(...)`。
- 上传与分析链路可复用已计算的 SHA-256，减少同文件多次全量读取。

3. 上传前预检链路统一为“双层快路径”
- 新增 `_check_upload_blacklist_precheck(...)`：先做黑名单快速拦截。
- 新增 `_run_upload_pre_risk_check(...)`：在黑名单未命中时执行“缓存命中 -> 模型检测”。
- 缓存命中时返回 `cache_hit=true`，直接复用历史风控结果，不再重复调用模型。

4. 三个上传入口已接入统一预检
- `/upload`：保持“黑名单优先于模型初始化”，命中后立即拒绝。
- `/upload_chunk_finalize`：合并文件后先黑名单，再缓存/模型风控。
- `/upload_batch`：每个文件均走统一预检链路，降低批量上传重复调用开销。

5. 分析链路补充“缓存复用 + 哈希复用”优化
- `process_video` 在黑名单比对与阻断登记时复用同一哈希，避免阻断场景下重复哈希计算。
- 在分析前风控阶段新增缓存命中逻辑：同文件且模型配置一致时可直接复用风控结果，减少重复视觉风控调用。

### 修改后的流程细节

#### A. 上传前风控新流程（单文件/分片完成/批量单文件）
1. 计算文件 `SHA-256`。
2. 黑名单比对（命中即 `403`，快速拒绝）。
3. 未命中时，按“文件哈希 + 模型配置签名”查询风控结果缓存。
4. 缓存命中：直接复用结果并继续后续上传流程。
5. 缓存未命中：调用模型风控并返回结果；若可缓存则写入缓存。
6. 若最终判定阻断：写入黑名单，后续同文件走更快拦截。

#### B. 性能收益点
1. 重复上传同一安全文件可跳过模型风控调用。
2. 黑名单命中与阻断登记场景减少重复哈希计算。
3. 批量上传场景下，重复文件更容易命中缓存，整体耗时更稳定。

#### C. 分析前风控复用流程（新增）
1. 分析开始先做黑名单指纹比对。
2. 未命中时优先查询风控结果缓存（同文件 + 同模型配置）。
3. 缓存命中则直接复用风险结果；未命中才执行视觉/文本风控。
4. 分析风控判定为可缓存结果时写入缓存，供后续分析/上传复用。

### 本次改动为达成的核心目标
- 目标 1：让上传前风控在“高重复文件场景”下显著降时延。
- 目标 2：保持现有风控拦截策略不变，仅优化执行路径。
- 目标 3：在不降低安全性的前提下，加快视频分析前置处理链路。

### 兼容性与注意事项
- 缓存命中条件是“文件内容一致 + 模型配置签名一致 + 策略签名一致”。
- 缓存不会替代黑名单；黑名单仍是最高优先级拦截路径。
- 缓存仅作为性能优化手段，不改变原有 `403/503` 风控语义。

## v2.0.4 (2026-03-16)

### 本版本目标
- 在现有风控判定之外，新增“违规视频指纹黑名单”机制，实现对完全一致违规视频的快速拦截。
- 将黑名单拦截前置到上传与分析入口，减少重复风控计算开销并降低复传绕过风险。
- 让黑名单数据可持续累积：对新命中的违规视频自动入库，后续同文件秒级拒绝。

### 已修改功能
1. 新增黑名单指纹库（SHA-256）

2. `upload_chunk` 阶段：
- `memory` 模式：分片写入进程内缓存。
- `disk` 模式：分片写入会话 `.part` 临时文件。
3. `upload_chunk_finalize` 校验分片完整性。
4. 内存模式在 finalize 一次性写入 `.staging`；磁盘模式则将 `.part` 移入 `.staging`。
5. 执行风控：
- 命中阻断：删除 `.staging` 文件并返回 `403`。
- 通过：移动到正式 `uploads`。

#### C. 分析流程
1. 分析前先做风控抽帧判定。
2. 命中阻断策略：分析请求直接失败返回 `403`（结构化 `risk`）。
3. 未命中：继续字幕提取、步骤分析、文档/PDF生成。

### 本次改动为达成的核心目标
- 目标 1：阻止违规视频进入正式存储区。
- 目标 2：在无人工复审情况下，实现可自动执行、可解释、可追踪的风控闭环。
- 目标 3：优化上传资源开销，减少分片阶段磁盘压力。
- 目标 4：提升前端告警可读性，降低排障成本。

### 兼容性与注意事项
- 内存模式在服务进程重启后不可恢复会话，需重新上传（会返回会话过期提示）。
- 若风控模型不可用，后端按“默认安全优先”策略拒绝上传/分析请求。
- 后续如需更严格策略，可调整 `restrict` 是否直接阻断及风险阈值。
