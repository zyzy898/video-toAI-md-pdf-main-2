# updata.md

## v2.0.4 (2026-03-16)

### 本版本目标
- 在现有风控判定之外，新增“违规视频指纹黑名单”机制，实现对完全一致违规视频的快速拦截。
- 将黑名单拦截前置到上传与分析入口，减少重复风控计算开销并降低复传绕过风险。
- 让黑名单数据可持续累积：对新命中的违规视频自动入库，后续同文件秒级拒绝。

### 已修改功能
1. 新增黑名单指纹库（SHA-256）
- 新增黑名单文件：`uploads/.risk_blocklist.json`。
- 新增指纹计算逻辑：基于文件内容计算 `SHA-256` 哈希。
- 新增黑名单读写与标准化逻辑（含版本、更新时间、命中/拦截计数）。

2. 上传入口新增黑名单前置拦截
- 单文件上传 `/upload`：文件进入 `.staging` 后先做指纹比对，命中即 `403` 拒绝。
- 分片上传 `/upload_chunk_finalize`：合并完成后先做指纹比对，命中即 `403` 拒绝。
- 批量上传 `/upload_batch`：每个文件单独做指纹比对，命中则该文件直接拒绝并记录错误信息。

3. 分析入口新增黑名单前置拦截
- `process_video` 在执行视觉/文本风控前，先执行黑名单指纹比对。
- 命中黑名单后，直接按拦截流程终止分析并返回结构化风控结果。

4. 风控命中后自动写入黑名单
- 当视频被风控判定为阻断（`block` 或按策略阻断）时，自动写入黑名单指纹。
- 后续再次上传/分析完全一致文件时，可直接命中黑名单并快速拦截。
- 返回风险详情中附带 `hash_sha256`，便于管理员审计与排查。

### 修改后的流程细节

#### A. 上传流程（新增黑名单前置）
1. 文件进入临时区（`.staging`）。
2. 计算文件 `SHA-256` 并比对黑名单。
3. 若命中：直接返回 `403`（`BLACKLIST_EXACT_HASH_MATCH`），不再进入模型风控。
4. 若未命中：继续执行既有视觉/文本风控流程。
5. 若后续风控命中阻断：将该文件哈希写入黑名单。

#### B. 分析流程（新增黑名单前置）
1. `process_video` 启动后先做黑名单指纹比对。
2. 若命中：立即拒绝分析并返回结构化风险信息。
3. 若未命中：继续执行现有风控检测与分析生成链路。
4. 若风控最终阻断：同步写入黑名单，供后续快速拦截。

#### C. 黑名单数据维护
1. 黑名单文件保存于 `uploads/.risk_blocklist.json`。
2. 记录字段包含：`sha256`、`reason_code`、`block_count`、`match_count`、时间戳等。
3. 命中黑名单时自动更新命中计数与来源信息，便于后续审计。

### 本次改动为达成的核心目标
- 目标 1：拦截完全一致违规视频，减少重复上传绕过。
- 目标 2：通过“前置指纹拦截”降低模型风控资源消耗。
- 目标 3：沉淀可追踪的违规样本指纹库，提升风控闭环能力。

### 兼容性与注意事项
- 该机制仅拦截“文件内容完全一致”的视频（精确哈希匹配）。
- 对剪辑、转码、重编码等非完全一致副本，哈希会变化，仍由原有风控模型判定。
- 黑名单文件建议纳入服务端备份与权限管理，避免被误删或篡改。
## v2.0.3 (2026-03-16)

### 本版本目标
- 在“主视觉风控异常”场景下，新增系统级视觉模型兜底，避免直接退化为单一文本判定。
- 在“视觉可用”场景下，同步接入文本兜底校验，采用更严格结果，降低漏检风险。
- 将风控抽帧策略升级为“随视频时长增长 + 分段随机分布”，提升长视频覆盖率并控制性能上限。
- 统一 `.env` 管理员提示为中文，减少运维理解成本并明确配置要求。

### 已修改功能
1. 主视觉不可用时新增 `.env` 视觉模型兜底
- 新增系统读取 `.env` 中的 `RISK_FALLBACK_API_KEY / RISK_FALLBACK_MODEL_NAME / RISK_FALLBACK_MODEL_BASE_URL`。
- 主视觉风控不可用（含不支持图片输入、模型服务异常、鉴权/配置受限）时，先自动尝试 `.env` 视觉模型。
- 仅当 `.env` 视觉模型也不可用时，才进入字幕关键词文本兜底链路。

2. 视觉可用场景新增文本二级兜底
- 当主视觉模型（或 `.env` 视觉兜底模型）可用并产出风险结果后，再执行一次字幕关键词风控。
- 合并策略采用“更严格优先”：若文本判定等级高于视觉判定，则覆盖最终决策。
- 文本链路新增非严格模式：当字幕信号不足时不单独触发拦截，避免对正常视频误伤。

3. 抽帧策略升级（时长驱动 + 随机分布）
- 新增视频时长探测（优先 `ffprobe`，失败时回退解析 `ffmpeg -i` 输出）。
- 抽帧数随视频时长增加：基础帧数从 `RISK_MAX_FRAMES` 起步，超过起始时长后按区间递增。
- 抽帧上限调整为 `15` 帧（`RISK_DYNAMIC_MAX_FRAMES=15`）。
- 时间点由固定列表改为“分段随机采样”，并使用稳定种子保证同一视频在同配置下可复现。

4. `.env` 提示与配置中文化
- `.env` 自动创建/补齐说明改为中文，明确“仅管理员使用”“必须是视觉模型”。
- 增加英文旧标记兼容，避免已存在旧版注释时重复追加提示块。

### 修改后的流程细节

#### A. 上传/分析风控执行链路（更新）
1. 先执行主视觉抽帧风控。
2. 若主视觉可用：得到视觉结果后，执行文本二级兜底并取更严格结果。
3. 若主视觉不可用：自动尝试 `.env` 视觉兜底模型。
4. 若 `.env` 视觉兜底可用：同样执行文本二级兜底并取更严格结果。
5. 若 `.env` 视觉兜底也不可用：退回文本兜底链路完成最终判定。

#### B. 动态抽帧流程（新增）
1. 先探测视频时长。
2. 根据时长计算目标抽帧数（长视频更多帧，最大 15）。
3. 将整段视频按帧数等分，每段内随机取样 1 帧。
4. 去重补齐后输出最终时间点列表，供主视觉与 `.env` 视觉兜底共用。

#### C. `.env` 提示与配置流程（更新）
1. 启动时自动检查 `.env` 是否存在视觉兜底配置块。
2. 若不存在则自动创建/补齐中文提示及关键配置项。
3. 该配置仅用于系统管理员维护，前端不引导普通用户填写。

### 本次改动为达成的核心目标
- 目标 1：将视觉风控从“单模型”提升为“主模型 + 系统兜底模型”的双层可用架构。
- 目标 2：在视觉可用时引入文本二级校验，提升高风险内容拦截覆盖率。
- 目标 3：通过时长驱动的动态抽帧提升长视频检测质量，并通过 15 帧上限控制成本。
- 目标 4：统一运维提示语言，降低配置与交接门槛。

### 兼容性与注意事项
- `.env` 视觉兜底模型必须支持图片输入，否则会自动回退到文本兜底。
- 动态抽帧会增加长视频检测耗时；当前已设置 15 帧上限控制资源占用。
- 文本二级兜底为“更严格优先”策略，理论上会提升拦截概率，后续可按阈值持续调优。

## v2.0.2 (2026-03-15)

### 本版本目标
- 在“视觉风控模型不支持图片输入”场景下，避免上传流程直接失败，提供可执行的第二套风控检测方案。
- 将字幕关键词风控从硬编码升级为可维护词库，支持持续迭代与快速调整。
- 继续优化前端上传体验：减少重复测试、增强检测阶段可见性、提升错误提示可读性与可操作性。

### 已修改功能
1. 第二套风控方案落地（视觉不可用时自动切换）
- 在上传风控中新增“模型不支持图片输入”识别逻辑。
- 命中该场景时，自动从“抽帧视觉检测”切换到“字幕关键词风控”兜底，不再直接返回配置失败。
- 字幕关键词风控输出与主风控一致的结构化结果：`decision/risk_level/reason_code/reason/scores/dimensions`。

2. 字幕关键词风控词库外置
- 新增 `risk_keyword_lexicon.json`，按 `nudity/violence/gore` 三类组织词库。
- 每类支持：`explicit`、`medium`、`reason_code_high`、`reason_code_medium`、`reason_label`。
- 后端新增词库加载与缓存逻辑，按文件修改时间自动刷新；若词库异常则自动回退默认内置词库，保证服务可用。

3. 安全策略加固（Fail-Closed）
- 当视觉模型不可用且字幕/文件名关键词信号不足时，默认按高风险拒绝上传，避免高风险内容漏拦。
- 保持“风控优先”原则，不因兜底启用而放宽拦截边界。

4. 上传前模型连通测试优化
- 上传前仍自动执行模型连通测试，但若用户“已测试通过且模型参数未变更”，则复用最近成功状态，跳过重复测试。
- 当自动测试触发且设置侧边栏未打开时，自动打开侧边栏并高亮引导“测试链接”按钮。
- 上传进度面板新增“安全检测”阶段提示，用户可感知风控执行过程。

5. 风控错误提示优化
- 优化 `risk_model_config_invalid / risk_model_auth_failed` 文案，弱化底层 `HTTP 400/401` 噪音，保留 `request id` 便于排障。
- 优化 `content_policy_violation` 对外提示，压缩冗长英文原文与分数碎片，输出可读的中文风险说明与处理建议。

### 修改后的流程细节

#### A. 上传风控执行链路（新增兜底）
1. 上传完成后进入风控检测（优先视觉抽帧）。
2. 若视觉模型可用：按既有视觉风控流程判定 `allow/restrict/block`。
3. 若视觉模型不支持图片输入：自动切换到字幕关键词风控。
4. 若字幕关键词信号充分：输出结构化风险结果并按策略拦截。
5. 若字幕关键词信号不足：按高风险拒绝上传（Fail-Closed）。

#### B. 字幕关键词风控词库加载流程
1. 启动/调用时读取 `risk_keyword_lexicon.json`（按 mtime 缓存）。
2. 对词库字段做标准化与去重处理（大小写、空值、非法码）。
3. 若外部词库缺失或格式异常，自动回退默认内置词库。
4. 运行时若文件更新，无需改代码即可使用新词库内容。

#### C. 上传前测试与提醒流程
1. 上传前自动校验模型连通性。
2. 若“参数未变且最近测试成功”，跳过重复测试直接上传。
3. 若触发自动测试且设置页关闭，自动打开设置侧边栏并引导至“测试链接”按钮。
4. 文件上传后展示“安全检测”阶段文案，直到风控完成。

### 本次改动为达成的核心目标
- 目标 1：在视觉模型能力受限时仍保持可执行风控，不中断拦截链路。
- 目标 2：将字幕关键词策略产品化为词库配置，降低后续维护成本。
- 目标 3：在保证安全前提下提升上传体验与提示可理解性。

### 兼容性与注意事项
- 词库文件为 UTF-8 编码，建议通过 `risk_keyword_lexicon.json` 统一维护，不要在代码中散落改词。
- 若外部词库内容异常，系统会自动回退默认词库，保证接口不因配置问题崩溃。
- Fail-Closed 会提高保守性：在证据不足场景可能增加拦截率，后续可通过词库和阈值继续调优。

## v2.0.1 (2026-03-15)

### 本版本目标
- 降低正常视频被误判拦截的概率，提升上传通过率与体验稳定性。
- 保持“风控优先拦截”原则，不因提示优化而弱化风控执行。
- 将“风控服务异常”与“真实违规命中”分离，避免错误状态被当成违规处理。
- 优化对外错误文案，不向用户暴露底层 API Key 配置细节。

### 已修改功能
1. 风控阈值与拦截策略优化（降低误杀）
- 调整阈值为更保守的拦截策略：`RISK_BLOCK_THRESHOLD=0.88`、`RISK_RESTRICT_THRESHOLD=0.68`。
- 调整 `RISK_BLOCK_ON_RESTRICT=False`，`restrict` 不再默认直接拦截上传。

2. 风控判定结果增加“校正层”
- 新增“分数优先校正”：模型文本返回 `block/restrict` 但分数不达阈值时自动降级。
- 新增“低置信度降级”：当 `confidence < 0.45` 时，判定自动降一级（`block -> restrict -> allow`）。
- 优化风控提示词：仅在“明确高风险证据”时才建议 `block`，减少不确定场景的过度拦截。

3. 风控异常与违规命中分流
- 新增风控基础设施异常识别（如模型不可用、抽帧失败、内部错误）。
- 上传接口遇到“风控服务异常”时返回 `503` 服务不可用，不再误报为违规 `403`。
- 只有真实命中风控策略时才返回 `403` + `content_policy_violation`。

4. 风控提醒文案优化（不暴露敏感配置细节）
- 统一对外提示为：`上传风控服务不可用，已拒绝上传。请联系管理员检查服务端风控配置后重试。`
- 不再在前端/接口文案中展示 `API Key 未配置` 等底层细节。
- 底层异常详情仅写入后端日志用于排障。

### 修改后的流程细节

#### A. 上传流程（单文件/分片/批量）
1. 文件进入 `uploads/.staging`（或分片 finalize 后进入 `.staging`）。
2. 执行风控检测并进行结果校正（阈值+置信度）。
3. 若判定为真实违规命中：返回 `403`，并拒绝上传。
4. 若判定为风控服务异常：返回 `503`，并给出统一服务不可用提示。
5. 若判定通过：移动至正式 `uploads`，上传成功。

#### B. 判定优先级（本次新增）
1. 先看数值分数决定基线等级。
2. 再融合模型文本 `decision`，若冲突则以安全但不过度的数值结果校正。
3. 最后基于 `confidence` 做一次降级保护，减少正常视频误杀。

### 本次改动为达成的核心目标
- 目标 1：在保持风控有效性的前提下，减少正常视频上传失败。
- 目标 2：避免将系统异常误判为违规，提升风控结果准确性与可解释性。
- 目标 3：优化用户侧提示信息，减少不必要的技术细节暴露。

### 兼容性与注意事项
- 本次并未关闭上传风控；风控仍为上传前置必经环节。
- 若风控服务不可用，上传将被拒绝（`503`），但不会误显示为违规命中。
- 如需更严格或更宽松策略，可继续调整阈值与 `restrict` 拦截策略。

## v2.0.0 (2026-03-15)

### 本版本目标
- 在“无人工复审”前提下，把内容风控前置到上传与分析主链路，命中高风险内容立即拒绝。
- 确保触发风控的视频不进入正式 `uploads` 目录，降低违规内容落库风险。
- 降低分片上传阶段磁盘写入频率，优先使用内存缓存，减少 I/O 开销。
- 让前端可读化展示风控结果，提升失败原因可解释性。

### 已修改功能
1. 后端自动风控分级（allow/restrict/block）
- 在后端新增风控判定与归一化逻辑，统一输出 `decision`、`risk_level`、`reason_code`、`reason`、`scores`。
- 支持阈值分级：`allow` / `restrict` / `block`，并按策略决定是否直接拦截。

2. 上传入口风控前置（拒绝后不入 uploads）
- 单文件上传 `/upload`：先写入 `.staging`，风控通过后才移动到 `uploads`。
- 分片上传 `/upload_chunk_finalize`：合并后先进入 `.staging`，风控通过后才进入 `uploads`。
- 批量上传 `/upload_batch`：每个文件都先风控，命中后直接拒绝。
- 风控拒绝统一返回 `403`，并包含 `code=content_policy_violation` 与 `risk` 详情。

3. 分片上传“内存优先”策略
- 为分片会话增加 `storage_mode`（`memory` / `disk`）。
- 小文件优先走内存缓存（默认单文件上限 64MB，全局上限 256MB）。
- 超出内存阈值自动回退到磁盘 `.part` 文件模式。
- 内存模式在 `finalize` 时一次性写入 `.staging`，减少中间落盘。
- 会话结束/删除时释放内存预留，避免资源泄漏。

4. 分析链路风控拦截
- 在 `process_video` 主流程前置风控检测。
- 命中阻断策略时，分析接口直接返回 `403`，并携带结构化 `risk` 信息。
- 批量分析中，单文件命中风控会返回该文件失败信息，不影响其他文件继续处理。

5. 前端风控结果提示接入
- 前端统一解析接口错误中的 `code` 与 `risk` 字段。
- 单个任务失败、批量任务失败均可展示风险等级/规则码/原因/分值摘要。
- 保证用户看到的是可读、可追踪的失败提示，而不是泛化报错。

### 修改后的流程细节

#### A. 单文件上传流程
1. 客户端发起上传请求。
2. 服务端先将文件写入 `uploads/.staging`。
3. 调用自动风控模型进行分级判断。
4. 若命中阻断策略：立即删除暂存文件，返回 `403` + 风控详情。
5. 若通过：将文件移动到正式 `uploads`，返回上传成功。

#### B. 分片上传流程（内存优先）
1. `upload_chunk_init` 创建会话并确定 `storage_mode`。
2. `upload_chunk` 阶段：
- `memory` 模式：分片写入进程内缓存。
- `disk` 模式：分片写入会话 `.part` 临时文件。
3. `upload_chunk_finalize` 校验分片完整性。
4. 内存模式在 finalize 一次性写入 `.staging`；磁盘模式则将 `.part` 移入 `.staging`。
5. 执行风控：
- 命中阻断：删除 `.staging` 文件并返回 `403`。
- 通过：移动到正式 `uploads`。

#### C. 分析流程
1. 分析前先做风控抽帧判定。
2. 命中阻断策略：分析请求直接失败返回 `403`（结构化 `risk`）。
3. 未命中：继续字幕提取、步骤分析、文档/PDF生成。

### 本次改动为达成的核心目标
- 目标 1：阻止违规视频进入正式存储区。
- 目标 2：在无人工复审情况下，实现可自动执行、可解释、可追踪的风控闭环。
- 目标 3：优化上传资源开销，减少分片阶段磁盘压力。
- 目标 4：提升前端告警可读性，降低排障成本。

### 兼容性与注意事项
- 内存模式在服务进程重启后不可恢复会话，需重新上传（会返回会话过期提示）。
- 若风控模型不可用，后端按“默认安全优先”策略拒绝上传/分析请求。
- 后续如需更严格策略，可调整 `restrict` 是否直接阻断及风险阈值。
